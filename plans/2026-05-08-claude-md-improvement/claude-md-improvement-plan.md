# 부록 B — 추가 5개 섹션 마크다운 초안 (그대로 붙여넣기 가능)

## 디버깅 프로토콜

버그·실패 시 표면 증상 가리기 전에 "왜 일어났나" 부터 답한다. 답이 "모름" 이면 추측 fix 금지 — 더 조사하거나 사용자에게 보고.

진행: **재현 가능한 최소 케이스 → 가설 1개 → 1개만 변경 → 결과 확인 → 가설 갱신**. 한 번에 여러 변수 바꾸지 말 것.

### 안티패턴 (감지 즉시 멈춤)
- `try/except`·`try/catch` 로 에러 삼키기
- 테스트 skip / xfail / `.only` / 주석 처리로 회피
- hardcoded fallback 으로 우회
- 같은 방향 fix 가 2회 실패하는데 계속 시도

### 왜
증상만 잡으면 같은 버그가 다른 형태로 재발. 추측성 패치는 코드와 컨텍스트를 동시에 오염시킨다.

---

## 변경 위생 — 단순·좁게

- **마무리 시점**: 같은 변경 안에서 "더 단순한 형태가 있나" 한 번 점검 (`/simplify` 활용 가능). 1곳뿐인 추상화, 중복, 죽은 코드, 불가능한 케이스 방어 코드 정리. 동작 보존 한도 안에서만.
- **작업 도중**: 요청 외 발견한 문제는 보고만 하고 별도 작업으로. drive-by 리팩토링·스타일·import 정리·의존성 교체 금지.

### 왜
처음 구현은 동작에 집중하므로 자연스레 늘어남 — 직후 단순화가 가장 싸다. 작은 요청이 예측 불가 변경 묶음으로 커지면 리뷰·롤백 비용 폭증.

---

## 리뷰 게이트: Plan / Push

큰 작업에는 두 단계 게이트.

1. **Plan reviewer** (Plan 직후 / 사용자 승인 전): 빠진 단계, 잘못된 가정, 더 단순한 대안, 위험·롤백, 영향 범위, 비범위(non-goals) 점검. subagent 가용 시 별도 컨텍스트 reviewer 에 의뢰. 1~2 파일 변경엔 생략.
2. **Code reviewer** (큰 변경 완료 / push 직전): correctness · tests · security · maintainability · impact 5관점. `/review` 또는 (push 직전) `/push-review`. "고칠 것 vs 잔여 리스크" 분리 기록.

### 왜
잘못된 plan 으로 50% 진행 후 되돌리는 비용 >> 1회 리뷰 비용. 작성자 관점에 갇히는 self-blind spot 차단.

---

## 불확실성·가정 명시

거짓 자신감 금지. 모르는 건 모른다고, 추측이면 추측이라고 표시.

- "확실히 X 일 것" 단언은 검증된 사실에만. 미검증은 "추정 / 가능성 / 코드 안 봤음" 표시.
- 사용자 요구가 모호하면 추측 진행 대신 **질문 1~2개로 좁히기** (사소한 결정까지 묻지는 말 것).
- 추천 함수·플래그·옵션의 실존이 불확실하면 Grep / WebSearch 로 확인 후 제시.

### 왜
거짓 자신감의 비용 > 잘못된 답의 비용. 신뢰가 한 번 깨지면 모든 응답에 검증 비용이 붙는다.
